## Enfoque de este capítulo

### Lo que no es (hoy)

❌ Casos con nombre de cliente, logo y métricas auditadas.  
❌ Testimonios inventados que suenen a marketing falso.  
❌ Rediseñar el capítulo cuando llegue el primer cliente.

### Lo que sí es

✅ **Casos de transformación** — Antes / Después / KPIs / Historia.  
✅ Estructura **MTX-CASE-00X** alineada a sectores MCM-05.  
✅ Plantilla lista para **reemplazar** el genérico por cliente real con datos verificables.  
✅ Evidencia social para objeciones (OBJ-001 · OBJ-004 · OBJ-008), demo, folleto y landing.

**Regla:** cada caso genérico debe poder convertirse en caso real **sin cambiar la estructura** — solo los campos marcados como *reemplazables*.

## Nivel de evidencia

No todos los casos tienen el mismo peso en conversación. El vendedor debe saber **cuánto puede afirmar**.

| Nivel | Estado | Qué puede decir el vendedor |
|-------|--------|----------------------------|
| **A** | Cliente documentado + métricas + cita | Nombre, logo, cifras y cita textual del cliente |
| **B** | Cliente documentado sin cifras públicas | Nombre e historia verificada — sin números en público |
| **C** | Caso validado internamente | Operación real conocida por el equipo — sin nombre público |
| **D** | Caso genérico | *«Empresas con operaciones como la suya…»* — sin atribuir cliente |

**Hoy:** todos los MTX-CASE-001–006 están en nivel **D**.

**Progresión:** D → C (post-trial interno) → B (cliente autoriza nombre) → A (métricas + cita publicables).

| Campo | Contenido | ¿Reemplazable con cliente real? |

| **Antes** | 3 bullets — dolor operativo | Sí — con datos reales |
| **Después** | 3 bullets — con Roustix | Sí — con métricas |
| **KPIs** | 3 indicadores del sector | Sí — valores reales |
| **Historia** | Párrafo narrativo | Sí — cita del cliente (nivel A) |
| **Cross References** | OBJ · MCM · PLAY · Landing · Folleto | Actualizar enlaces |
| *Cliente* | *(vacío hoy)* | Nombre · ciudad |
| *Plan* | *(vacío hoy)* | Start / Grow / Scale |
| *Métricas* | *(vacío hoy)* | % mejora · tiempo ahorrado |

### Checklist · Migrar a caso real

1. Validar con el cliente permiso de uso (nombre, logo, cifras).
2. Subir **nivel de evidencia** (D → C → B → A).
3. Registrar **fecha de publicación**.
4. Completar **empleados · industria · país** para matching con prospectos.
5. Reemplazar **Historia** con cita o narrativa verificada.
6. Añadir **Métricas** con números reales (nivel A).
7. Actualizar **Cross References** en OBJ, PLAY, landing y folleto.
8. Registrar en Debrief qué caso usó el vendedor.

## Reflexión · Sprint 5

Cuando empezamos a conversar sobre Roustix hablábamos de Flask, módulos, bases de datos y dashboards.

Hoy estamos definiendo procesos de marca, diseño, experiencia de usuario, ventas y adopción.

Si alguien leyera únicamente toda esta documentación, probablemente asumiría que Roustix ya tiene un equipo de producto, diseño, UX y comercial trabajando detrás.

Y, en cierto modo, ese era el objetivo desde el principio: construir una plataforma que no solo funcione técnicamente, sino que también tenga una **identidad** y una forma **consistente** de crecer.

Ese trabajo ahorrará muchos problemas cuando empieces a incorporar más personas al proyecto.

**Los casos de transformación son el puente:** hoy narran lo posible; mañana documentan lo logrado — con la misma ficha.
